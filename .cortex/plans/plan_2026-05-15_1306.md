## Context

I explored the full-stack fraud intelligence platform and identified **3 distinct bugs** causing the reported issues:

### Issue 1: Transaction page shows no data
**Root cause**: The `_generate_transaction()` function in `backend/app/routers/transactions.py` creates `TransactionResponse` objects but **never sets required fields**: `customer_name`, `merchant_category`, and `country`. These are required (no defaults) on the `TransactionResponse` Pydantic model. This causes a Pydantic `ValidationError` on every request to `GET /api/transactions`, returning a 500 error to the frontend. The frontend's `useTransactions` hook gets an error, so `data` is undefined and the table shows empty/loading state.

### Issue 2: Alerts not appearing  
**Root cause**: Same pattern — `_generate_alert()` in `backend/app/routers/fraud_alerts.py` creates `FraudAlertResponse` objects but **never sets required fields**: `customer_name`, `country`, `channel`, and `created_at`. Additionally, it passes `category` and `description` which don't exist on the schema. This causes validation errors on `GET /api/alerts`, so the REST-based alert feed fails.

### Issue 3: Chat returns "alert_id required" error
**Root cause**: The `InvestigationRequest` Pydantic model (`schemas.py:92-94`) defines `alert_id: str` as required and has no `message` field. But the frontend sends `{ message: "...", context: {...} }` to `POST /api/investigation/chat`. The route handler at `investigation.py:19` also references `body.message` which doesn't exist on the model. The response model `InvestigationResponse` also mismatches — it expects `investigation_id`, `alert_id`, `status`, `created_at`, `updated_at` but the handler returns `InvestigationResponse(data=...)` which isn't a valid field.
